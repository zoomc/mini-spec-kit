# mini-speckit

**别再让 AI 代理不写计划就写代码了。**

`mini-spec` 是一个轻量级的规范驱动开发框架，强制 AI 编码代理在动手之前先思考。这是防止 AI 跑偏、引入 Bug 或构建错误东西的最小可行流程。

## 问题是什么？

AI 编码代理很强大，但很急躁。给它一个任务，它会立即开始写代码——通常在理解你真正想要什么之前。结果呢？

- ❌ 代码不符合你的需求
- ❌ 无休止地来回修复本不该存在的问题
- ❌ 没有文档记录构建了什么或为什么构建
- ❌ 范围蔓延和功能漂移

## 解决方案

`mini-spec` 强制执行一个简单的 6 阶段工作流：

```
规格说明 → 规划 → 写检查清单 → 分析 → 实现 → 对账
```

**在写任何代码之前**，代理必须：
1. ✅ 定义具有可追溯 ID 的需求
2. ✅ 创建实施计划
3. ✅ 编写验证检查清单
4. ✅ 分析一致性及缺口
5. ✅ 只有此时才写代码
6. ✅ 验证并更新规格以匹配现实

**规格文档是真实的来源。代码服务于规格。**

## 为什么有效？

| 原则 | 好处 |
|------|------|
| **规格为王** | 代码实现规格，而不是反过来 |
| **工件门控** | 每个阶段产生下一个阶段所需的文件——跳过阶段就无法继续 |
| **无预选框** | 每个验证都从 `[ ]` 开始，强制实际测试 |
| **变更追踪** | 每个修改都用需求 ID 记录 |
| **对账强制** | 编码后必须更新规格以反映现实 |

## 6 阶段工作流

| 阶段 | 发生什么 | 输出 |
|------|----------|------|
| **1. 规格说明** | 定义要构建的内容，带需求 ID | `spec.md` 带 `<!-- REQ-XXX -->` 标记 |
| **2. 规划** | 分解为带依赖关系的任务 | `plan.md` 带 `## 任务列表` |
| **3. 检查清单** | 为每个任务创建验证命令 | `checklist.md` 带 `[ ]` 项 |
| **4. 分析** | 只读一致性检查 | `plan.md` 带 `## 分析报告` |
| **5. 实现** | 按计划编写代码 | 可运行的代码 + 通过测试 |
| **6. 对账** | 验证所有内容，更新规格 | `checklist.md` 全部 `[x]` + `changelog.md` |

## 阶段门控（防跳过机制）

每个阶段都有一个**物理门控**——一个必须通过的文件检查，然后才能进入下一阶段：

```
规格说明 → [spec.md 有 ## 需求？] → 规划
规划 → [plan.md 有 ## 任务列表？] → 检查清单
检查清单 → [checklist.md 有 [ ] 项？] → 分析
分析 → [plan.md 有 ## 分析报告？] → 实现
实现 → [构建/测试通过？] → 对账
对账 → [全部 [x]，规格匹配代码？] → 完成
```

**跳过门控？工作流停止。** 没有捷径，没有"以后再修"。

## 快速开始

### 在你的项目中初始化

```bash
git clone https://github.com/zoomc/mini-speckit.git
cd mini-speckit
./scripts/init-mini-speckit.sh /path/to/your/project
```

这会将 `.mini-speckit/` 模板复制到你的项目中。

### 与 AI 代理一起使用

1. 你的 AI 代理读取 `.mini-speckit/project-constraints.md`
2. 为每个任务遵循 6 阶段工作流
3. 产生可追溯、可验证的结果

## 目录结构

```
your-project/
  .mini-speckit/
    project-constraints.md    # 工作流规则和约束
    project-spec.md           # 项目目标和数据流
    modules/
      <module-name>/
        spec.md               # 当前需求（带 REQ-XXX ID）
        plan.md               # 实施计划 + 分析报告
        checklist.md          # 带命令的验证项
        complement.md         # 模块职责
        changelog.md          # 变更历史
```

## 变更管理

| 类型 | spec.md | changelog.md |
|------|---------|-------------|
| Bug 修复 | 无更改（或 `## 已知问题`） | 记录修复 |
| 需求变更 | 标记 `[CHANGED]` / `[DEPRECATED]` | 记录变更 |
| 增强 | 添加 `[NEW]` 部分 | 记录添加 |
| 重构 | 无更改（行为保持） | 记录重构 |

## 谁适合使用？

- **AI 编码代理**（Claude、GPT、Copilot、Cursor 等）——给它们一个结构化的工作流来遵循
- **使用 AI 的开发者**——确保 AI 输出符合你的需求
- **团队**——为每个变更创建文档轨迹
- **独立开发者**——防止你自己的范围蔓延

## 真实示例

**使用 mini-spec 之前：**
> "嘿 AI，构建一个用户认证系统"
> → AI 写了 500 行代码
> → 你发现它不处理密码重置
> → 来回折腾好几个小时

**使用 mini-spec 之后：**
> 阶段 1："需求：登录、登出、密码重置、邮箱验证"
> 阶段 2："计划：4 个任务，每个约 100 行，依赖关系已映射"
> 阶段 3："检查清单：8 个验证命令已就绪"
> 阶段 4："分析：未发现缺口"
> 阶段 5："实现：代码完全匹配计划"
> 阶段 6："对账：全部验证通过，规格已更新"

**总共节省的时间：数小时的返工。**
